# `7W1H`

Canonical north-star document for `@repo/audit`.

The file name stays explicit on purpose. If the audit contract drifts, this is the first place to check and the first place to tighten.
The code surface mirrors this file with `Audit7W1H*` types and `audit7w1h*` schemas.

## 7W1H Contract

Every governed audit event should answer these questions:

| Question | Fields | Why it matters |
| --- | --- | --- |
| Who | `actorId`, `actorType`, `actorRole` | Identifies who performed the action and the role they held in context. |
| Whom | `subjectType`, `subjectId` | Identifies the affected person or entity when the action targets one. |
| What | `action`, `summary` | Describes the governed mutation in stable terms. |
| When | `occurredAt` | Captures the authoritative event timestamp. |
| Where | `tenantId`, `companyId`, `module`, `surface`, `route` | Pins the event to the exact application context. |
| Why | `reason`, `policyReference`, `approvalId` | Records the business or policy basis for the action. |
| Which | `targetType`, `targetId` | Identifies the durable record or resource affected. |
| How | `channel`, `requestId`, `operationId` | Reconstructs the execution path that produced the event. `requestId` is generated when omitted, and `operationId` falls back to it. |

## Guardrails

- Keep audit as evidence, not observability.
- Keep request and operation identifiers stable enough to trace a single mutation end to end.
- Keep redaction bounded and deterministic.
- Keep summary and reason normalized so filters and exports stay searchable.
- Keep `requestId` stable across the execution pipeline; the writer generates it when the caller omits one.
- Keep success writes transaction-aware whenever the caller already owns a transaction.

## Result

If a mutation cannot be expressed cleanly through this contract, the mutation flow needs to change before the audit model does.
